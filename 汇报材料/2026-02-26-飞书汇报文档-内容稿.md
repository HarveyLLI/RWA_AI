## 一、业务简介

### 1.1 主营业务是什么

- 我们**继续做借贷业务**：客户有借贷需求，我们产生借贷订单（利率、期限等），需要放款资金。
- **债的转化**：将借贷需求通过智能合约标准化、代币化，变成可在链上交易的「债」类 DeFi 产品。
- **资金端**：N 个投资者在基金会运营的平台上**买债**，获得收益；**募资所得作为放款资金**，即募资 = 放款资金来源。
- **合规分工**：我们作为**技术服务方**，提供合约与平台（技术+风控）；由**基金会**采用我们的合约与平台，面向投资者募资；募资所得按 MLA 等安排，最终作为借款客户的放款资金。

### 1.2 为什么做 RWA（做这件事的好处）

| 对比项 | RWA 之前 | RWA 之后 |
|--------|----------|----------|
| **放款资金来源** | 自有资金，或单一渠道资金 | 通过「债」的 DeFi 产品，由 N 多投资者买债，募资作为放款资金 |
| **结果** | 资金规模受自有/单渠道约束 | **资金来源被拓宽** → **业务规模可以扩大** |

一句话：**RWA 不是不做借贷，而是把借贷订单对应的「债」通过合约变成 DeFi 产品，由基金会合规地向投资者募资，募来的钱给借款客户放款，从而拓宽我们的资金来源、做大业务规模。**

### 1.3 我们的新身份（主站定位）

- **技术赋能者 + 风控预言机**：我们不直接面向散户募资、不直接运营「卖债」的 DApp；我们提供合约与平台、风控与透明度能力，基金会用我们的技术对投资者募资与卖债。
- **官网**：确立新身份，展示技术能力、合作方、项目/风控透明度披露，不直接募资；若引导用户去基金会或合作 DApp，需合规弹窗（第三方运营、我们仅技术+风控）。

### 2.1 角色与职责简要说明

| 角色 | 职责 |
|------|------|
| **借款人** | NS 的客户；发起质押借贷，提供抵押品（如 BTC、ETH）。 |
| **债权工厂（Originator / NS）** | 每个项目的发起人、专业放贷人；接收质押借贷、持有抵押品；向 AA 平台提交准入与尽调；与基金会签订 MLA。 |
| **AA 平台（技术提供方）** | 前端网站、每笔债的尽调与风险管理、风险数据上预言机、储备 Oracle PoR、反洗钱、KYC 工具包（链上 DID）；审核(ASA)后按 MLA 分配资金给离岸主体。 |
| **AA 平台（智能合约开发方）** | 多币种结算、自动化 NAV、预留 AI-Agent Kit、构建项目资产池；向基金会侧提供服务。 |
| **离岸运营主体（基金会）** | 类基金管理人 & 智能合约运营者；链上募资的募资人，实际掌握资金金库；与发起人(NS)签 MLA，接收分配资金；面向 Web3 用户、机构、AI Agent 募资与卖债。 |
| **Web3 用户 / 机构** | 真正出资的投资人；向基金放贷 USDT、获得 LP token、获得股票 Token；可在股票 Token 二级市场交易。 |
| **AI Agent** | 与 Fund Manager 交互，参与资金管理或策略执行；未来「找收益」的 Agent 可发现、读懂、推荐、使用我们的产品与数据。 |

| 亮点模块 | 功能要点 |
|----------|----------|

- **被 Agent 发现**：官网与平台对 Agent 友好（结构化数据、API/文档、清晰的产品与风险说明）。

## 五、AI 在过程中的支持

### 5.1 对内（我们内部）

| 环节 | AI 能做什么 |
|------|-------------|
| **尽调与定价** | 辅助评估抵押品质量、建议 LTV/利率/期限。 |
| **合同与结构** | 辅助生成或校验合同条款、债的结构。 |
| **实时风控** | 持续监控 LTV、抵押品价值、市场波动；接近清算或异常时告警。 |
| **合规与审计** | 异常检测、KYC/AML 辅助、审计日志摘要。 |
| **产品推广** | 用 AI 写推广文案、落地页、产品与能力介绍、合规话术、多语言素材，把新产品推广出去，让更多人与 Agent 发现我们。 |

### 5.2 对客户

| 对象 | AI 能做什么 |
|------|-------------|
| **借款人** | 产品与流程说明、自助查询（抵押率、建议操作）、申请引导；智能客服。 |
| **投资者** | 产品与风险解释；通过**自己的 AI Agent** 监控、告警、在授权下操作；智能客服；**被 Agent 发现**（官网与平台对 Agent 友好，让「找收益」的 Agent 能发现、读懂、推荐我们）。 |
| **合作方/开发者** | 接入与调试支持、API/MCP/Key 与文档；用 AI 生成或辅助的推广与产品说明。 |

### 5.3 老板思路与产品设计

- **趋势**：越来越多人会通过 **AI Agent 打理资金**，Agent 会**自己找赚钱的项目**。  
- **我们的应对**：产品与数据要**对 Agent 友好**（可发现、可读懂、可推荐、可使用）；用 **AI 做推广**，让更多人与 Agent 发现我们；**智能客服**提升体验与效率。  
- **在网站中的体现**：官网开发者/AI 门户、基金会平台「用 AI Agent 连接」入口、全站智能客服与自助、AI 生成的推广与产品文案。

## 六、分阶段目标

### 6.1 阶段一：需求与资金方验证（建议优先）

- **目标**：验证「谁愿意出钱、在什么条件下出钱」。  
- **关键动作**：小规模、非链或半链试点；1～2 家意向资金方/机构访谈或 pilot；明确利率、期限、披露与合规形式。  
- **产出**：资金方意愿与条件结论；对产品与披露的修正建议。

### 6.2 阶段二：合规与结构

- **目标**：确保基金会、MLA、链上募资的可行性与法务/合规落地。  
- **关键动作**：与法务/合规对齐 RWA 免责、离站弹窗、话术规范；确认链上募资与 DID/KYC 方案。  
- **产出**：合规与结构可执行方案；官网与基金会平台合规话术与披露标准。

### 6.3 阶段三：产品与平台规模化

- **目标**：官网新导航与内容、基金会平台、AI（智能客服、推广、被 Agent 发现）完整落地，可交付开发并走向生产。  
- **关键动作**：按「网站亮点功能清单」与「设计/开发必含项」完成设计与 demo；交接开发，支持上线与迭代。  
- **产出**：可演示的官网（含新导航）与基金会平台 demo；需求与设计文档齐全，可交接开发。

### 6.4 与当前汇报的关系

- **本次汇报**：呈现完整业务简介、角色与职责、双站关系、亮点清单、AI 支持、分阶段目标；**网站 demo 作为其中一环**展示。  
- **后续**：按阶段一 → 二 → 三 推进，或在老板确认后调整阶段顺序与优先级。
