# 两个网站项目简述

> 整理日期：2026-02-26  
> 用途：明确两个网站各自的内容与职责，便于需求与设计时对齐。

---

## 一、两个项目分别是什么

| 项目 | 名称/主体 | 简要定位 |
|------|-----------|----------|
| **项目一** | **Antalpha 官网**（含新增导航及该入口下的全部内容） | Antalpha 品牌与能力展示；新增一个导航入口，专门呈现「技术服务 & 风控服务」相关能力与透明度。 |
| **项目二** | **基金会平台** | 基金会运营的**线上平台**，用于面向投资者的**产品发布**与**链上募资**（投资者在此买债、获得收益）。 |

---

## 二、项目一：Antalpha 官网（含新导航）

**内容与职责：**

- **主体**：Antalpha（我们）。
- **主要职责**：
  - 确立「技术赋能者 + 风控预言机」的身份。
  - **介绍我们的技术能力**（合约、风控、透明度能力等）。
  - **展示相关合作方**（如 Powered by Antalpha Prime 生态、合作 DApp/基金会等）。
  - **项目/风控透明度的披露**（如全局指标、资金池透视、托管证明等，证明底层资产真实、安全、可验证）。
- **合规与导流**：不直接卖债/募资；若引导用户去基金会或合作 DApp，需合规弹窗（如「第三方运营，我们仅技术+风控」、用户同意后再跳转）。
- **新导航**：在官网主导航新增一个入口，名称为「技术服务 & 风控服务」产品名（待定）；进入该入口后的全部内容与体验由您设计。

**一句话**：官网 = 品牌 + 技术能力 + 合作方 + 透明度披露，不直接募资。

---

## 三、项目二：基金会平台

**内容与职责：**

- **主体**：基金会（采用我们合约与平台、面向投资者募资的一方）。
- **主要职责**：
  - **真正的产品发布**：将「债」类产品（RWA DeFi 产品）对外发布，展示池子、APY、风险说明等。
  - **链上募资**：投资者在此参与买债/投资，募资所得作为放款资金。
- **与我们的关系**：我们提供**合约与平台（技术+风控）**；基金会运营该平台、面向投资者募资与卖债。
- **与官网的关系**：官网侧重能力与透明度；基金会平台侧重**面向投资者的产品与募资**。两者在合规与导流上衔接（如官网「合规流量分发器」可导流至基金会平台，并配合离站弹窗）。

**一句话**：基金会平台 = 面向投资者的产品发布 + 链上募资（买债）。

## 四、对照小结

| 维度 | 项目一：Antalpha 官网 | 项目二：基金会平台 |
|------|------------------------|----------------------|
| **主体** | Antalpha | 基金会 |
| **主要用户** | 想了解我们能力、合作方、透明度的访客；验证资产安全的投资者；开发者/合作方 | 要买债、获收益的投资者 |
| **核心职责** | 技术能力介绍、合作方展示、项目/风控透明度披露 | 产品发布、链上募资（投资者买债） |
| **是否直接募资** | 否 | 是 |
| **我们提供** | 官网内容与新增导航下的设计；技术能力与数据 | 合约与平台（技术+风控） |

## 五、AI 相关内容在哪个平台？

| AI 相关能力 | 归属平台 | 说明 |
|-------------|----------|------|
| **开发者与 AI-Agent 门户**（API 文档、MCP、接口清单、快速启动代码） | **Antalpha 官网** | MRD 模块三明确为「Prime 主站」升级内容；技术能力由我们提供，文档与接入入口放在官网（新导航下）更合理。 |
| **API Key 生成与管理、API 配置** | **Antalpha 官网** | 属于「我们提供的技术能力」，开发者/AI Agent 在此申请 Key、配置，逻辑上归属官网。 |
| **智能合约库、审计报告** | **Antalpha 官网** | 同上，技术能力展示与下载放在官网。 |
| **客户沟通、智能客服**（借款端/合作方/开发者咨询） | **Antalpha 官网** | 面向「了解能力、接入技术」的访客与开发者，官网承载。 |
| **客户沟通、智能客服**（投资者端） | **基金会平台** | 面向「买债、查持仓、验证资产」的投资者，在基金会平台提供。 |
| **「用 AI Agent 连接」的入口或说明**（投资者用 Agent 监控/下单） | **基金会平台** | 投资者在基金会平台操作，可在此提供「用你的 AI Agent 连接我们」的入口或引导；实际 API 文档与 Key 申请仍在官网。 |

**小结**：  
- **Antalpha 官网**：承载 **AI 技术能力与接入**（开发者/AI-Agent 门户、API/MCP/Key、文档、智能合约库）以及面向合作方/开发者的**客户沟通与智能客服**。  
- **基金会平台**：承载投资者端的**智能客服**，以及**「用 AI Agent 连接」的入口或说明**（能力由我们提供，入口在投资者使用的平台）。

## 六、AI 相关内容的目标群体分别是谁？

### 项目一（Antalpha 官网）— AI 内容的目标群体

| 目标群体 | 说明 |
|----------|------|
| **基金会（作为技术接入方）** | 基金会采用我们的合约与平台运营募资端，其技术团队可能需要对接我们的 API（获取透明度数据、风控数据、池子信息等），因此基金会是官网 AI/API 内容的**目标群体之一**。 |
| **AI Agent 开发者** | 需要标准化 API 和 MCP 来开发「读取收益、执行投资」的 Agent 的人；他们来官网看文档、申请 Key、拿示例代码。MRD 明确列出的核心目标群体。 |
| **DApp 运营商 / 离岸基金** | 寻找合规技术底座的一方，可能用我们的 API 或 AI 能力做集成、自动化，也属于**技术接入方**。 |
| **真正出资的投资人（其中「用 AI Agent 对接」的那部分）** | 若投资者自己或通过团队要用 AI Agent 监控、下单，他们可能来官网**申请 API Key、查文档**；官网 AI 内容也服务这类用户，但官网整体更偏「技术能力与接入」，主力受众仍是**技术接入方**（基金会、开发者、机构），而非「只来买债的散户」。 |

**一句话（项目一）**：官网 AI 内容主要面向**技术接入方**——基金会（技术侧）、AI Agent 开发者、DApp/机构；真正出资的投资人里「用 Agent 对接」的那部分也会用到，但官网不主打散户投资者。

**重要纠正与老板思路**（详见《项目一AI用途纠正与老板思路及AI推广》）：
- 我们合作的基金会都是**线下 KYC**，不是随便就能接入的，因此项目一里「对基金会开放接入」的 AI 用途**有限**。
- 老板思路：未来很多人会通过 **AI Agent 打理资金**，Agent 会**自己找赚钱的项目** → 要从「**投资者的 Agent 主动找收益、发现我们**」的角度想：官网/平台要**对 Agent 友好**（可发现、可读懂、可推荐、可使用），而不是主打「开放给基金会申请接入」。
- **AI 用于产品推广**：用 AI 写推广文案、做产品与能力介绍、合规话术等，把新产品推广出去，让更多人和更多 Agent 发现我们；项目一、项目二都可使用 AI 做推广内容。

---

### 项目二（基金会平台）— AI 内容的目标群体

| 目标群体 | 说明 |
|----------|------|
| **真正出资的投资人** | 在基金会平台买债、获收益的人。平台上的智能客服、「用 AI Agent 连接」的入口与说明，都是为**这些投资者**服务的——让他们查持仓、问问题、用自己的 Agent 监控/告警/在授权下操作。 |
| **基金会** | 是平台**运营方**，不是 AI 内容的「目标群体」；目标群体是**使用平台的投资者**。 |

**一句话（项目二）**：基金会平台 AI 内容的目标群体就是**真正出资的投资人**（在平台买债、需要客服与 Agent 连接入口的那批用户）。

### 对照小结

| 项目 | AI 内容目标群体 |
|------|-----------------|
| **项目一：Antalpha 官网** | **基金会（技术侧）**、**AI Agent 开发者**、**DApp/机构**；以及真正出资的投资人里**用 AI Agent 对接**的那部分（偏技术接入方）。 |
| **项目二：基金会平台** | **真正出资的投资人**（买债、查持仓、用客服、用 Agent 连接入口）。 |
