## 一、业务图整体结构（从左到右）

```
借款人 → 债权工厂(Originator) → 质押抵押品 + 准入/尽调 → AA平台(技术提供方+合约开发方)
                                                                  ↓ 审核(ASA)、分配资金(MLA)
离岸运营主体(类基金管理人&智能合约运营者) ← 分配资金(MLA)
         ↓
    AI Agent、Web3用户、机构、股票Token二级市场
```

## 二、各角色与职责（按图）

### 1. 借款人（Borrower — NS 的客户）

- 发起**质押借贷（Pledged Lending）**，与债权工厂交互。
- 提供**质押各类抵押品**（如 BTC、ETH）。

### 2. 债权工厂（Credit Factory — Originator）

- **每个项目的发起人**，**以 NS 为代表的专业放贷人**（NS = 我们/Antalpha 侧）。
- 接收借款人的质押借贷，**持有**质押的各类抵押品（如 BTC、ETH）。
- 向 AA 平台的技术提供方提交**准入、尽职调查**信息。
- 与离岸基金**签订 MLA**（基金与发起人(NS)签订 MLA）。

### 3. 质押抵押品（Pledged Various Collaterals）

- 持有 BTC、ETH 等抵押品。
- 向 AA 平台技术提供方提供**监控（Monitoring）**数据。

### 4. AA 平台（AA 平台 — AA 相关主体 A (BVI)）

图中为我们技术侧的主体，内部分为两块：

#### 4.1 技术平台的提供者（Technology Platform Provider）

- **前端网站开发**
- **每个债权（借贷）项目的尽职调查**
- **每个项目的风险管理**
- **风险数据透明化上预言机**
- **储备 Oracle PoR（Proof of Reserve）**
- **反洗钱工具包**
- **KYC 工具包（链上 DID）**
- 接收债权工厂的**尽调**与抵押品的**监控**数据，进行**审核（ASA）**后，将**分配资金（MLA）**给离岸运营主体。

#### 4.2 智能合约的开发者（Smart Contract Developer）

- **多币种结算**
- **自动化净值计算（NAV）**
- **预留 AI-Agent Kit**
- **构建项目资产池**
- 向离岸运营主体的「类基金管理人 & 智能合约运营者」**提供服务**。

### 5. 离岸运营主体（Offshore Operating Entity — VA 基金 / crypto fund）

- 内部角色：**类基金管理人 & 智能合约的运营者**。
- **链上募资平台的募资人**，**实际掌握资金金库**。
- 与发起人（NS）**签订 MLA**，接收**分配资金（MLA）**。
- 对外交互：
  - **AI Agent**：与 Fund Manager 直接交互（AI 参与资金管理/策略执行）。
  - **Web3 用户**：向基金**放贷 USDT**，**获得 LP token**，从基金**获得股票 Token**。
  - **机构**：与 Fund Manager 交互（投资或合作）。
  - **股票 Token 二级市场**：基金可**买入股票**；Web3 用户获得的股票 Token 可在此二级市场交易。

## 三、与既有业务资料的对应关系

| 图中角色/概念 | 我们已讨论的对应 |
|---------------|------------------|
| **债权工厂（Originator）/ NS** | 我们的借贷业务端：客户借贷需求 → 我们产生借贷订单、持有/管理抵押品；「债」的发起人。 |
| **AA 平台（技术提供方 + 合约开发方）** | Antalpha Prime 作为**技术赋能者与风控预言机**：提供技术平台、尽调、风控、预言机、PoR、KYC/AML、智能合约（含**预留 AI-Agent Kit**）。对应**项目一（官网）**所展示的能力与透明度。 |
| **离岸运营主体 / 类基金管理人 & 智能合约运营者** | **基金会**：采用我们合约与平台，面向 Web3 用户与机构**链上募资**，实际掌握资金金库；对应**项目二（基金会平台）**的运营方。 |
| **分配资金（MLA）** | 募资所得按 MLA 分配给离岸主体，最终作为**放款资金**给到借款端。 |
| **Web3 用户 / 机构** | **真正出资的投资人**：放贷 USDT、获得 LP token/股票 Token，在二级市场可交易。 |
| **AI Agent** | 图中与 Fund Manager 直接交互，对应老板思路：**AI Agent 打理资金、找赚钱的项目**；我们侧在合约中**预留 AI-Agent Kit**，离岸侧由 AI Agent 与基金管理人交互。 |
| **KYC 工具包（链上 DID）** | 与 MRD「合规与 DID 工具箱」一致；图中为**链上 DID**，与「基金会线下 KYC」可并存（链上 DID 面向终端用户/合规展示）。 |

## 四、从图中提炼的要点（对产品与设计的影响）

1. **技术提供方（我们）的职责在图中很清晰**：前端网站、尽调、风控、预言机、PoR、AML、KYC（链上 DID）、审核（ASA）；合约侧：多币种结算、NAV 自动化、**AI-Agent Kit**、资产池构建。这些可与**项目一（官网）**的「技术能力、透明度披露」模块一一对应。
2. **AI 在图中两处出现**：  
   - **AA 平台 / 智能合约开发者**：**预留 AI-Agent Kit**（我们提供能力）；  
   - **离岸运营主体**：**AI Agent** 与 Fund Manager 直接交互（投资者/机构侧用 Agent 管理资金或执行策略）。  
   与「被 Agent 发现、被 Agent 使用」及「Agent 找收益」思路一致。
3. **链上 KYC（DID）**：图中技术提供方包含 **KYC 工具包（链上 DID）**，说明除线下基金会 KYC 外，平台侧有链上身份与合规能力，可与 MRD 合规与 DID 工具箱、ZKP 等结合设计。
4. **资金流**：借款人 → 债权工厂（抵押品）→ 我们尽调/风控/审核(ASA) → 分配资金(MLA) → 离岸基金 → Web3 用户/机构（募资、LP token、股票 Token）→ 二级市场；与「借贷需求 → 债 → 募资 → 放款」的叙述一致。
5. **项目一 vs 项目二**：图中 **AA 平台**对应我们官网与能力展示（项目一），**离岸运营主体 + 其前端/募资**对应基金会平台（项目二）；AI Agent 既使用我们提供的 Kit（项目一侧能力），也与 Fund Manager（项目二侧）交互。
