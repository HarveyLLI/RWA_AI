## 二、AI Agent 支持 — 可落地的设计思路（扩展 idea）

以下按「对外宣传」与「真实能力」两条线整理，便于后续选型与排期。

### 1. 身份与接入（您已提，略作展开）

| 方向 | 说明 | 可做内容 |
|------|------|----------|
| **API Key 生成与管理** | Agent 需要稳定、可追溯的凭证 | 控制台：创建 API Key、命名、绑定 IP/域名白名单、权限范围（只读/交易/风控）；支持多 Key（开发/生产/按 Agent 隔离）；Key 轮换与撤销。 |
| **API 配置中心** | 统一管理 Agent 可调用的端点与参数 | 文档化 Base URL、环境（testnet/mainnet）、限速与配额；可选：图形化配置「允许调用的接口清单」、webhook 回调地址。 |
| **权限与范围** | 合规与安全 | 按角色/Key 限制：仅查询（APY、LTV、NAV）、仅风控、或含交易；审计日志：哪个 Key 在何时调用了什么。 |

### 2. 协议与文档（与 MRD 中的 MCP 一致）

| 方向 | 说明 | 可做内容 |
|------|------|----------|
| **MCP（Model Context Protocol）** | 便于 AI 模型/Agent 标准化接入 | 提供 MCP server 或 MCP 兼容的接口说明；文档中明确：get_pool_apy()、verify_kyc_status(address)、get_realtime_ltv() 等与 MCP 的对应关系。 |
| **开发者/AI 门户** | 一处入口、人类与 AI 共用 | 接口清单、请求/响应示例、错误码；**快速启动代码块**（Python/TS）演示「Agent 如何调预言机/查询/下单」；可选：Postman/OpenAPI 导出、SDK。 |
| **「AI Agent Kit」品牌** | 对外噱头 + 真实入口 | 独立子站或官网区块：「AI Agent Kit」或「Build with AI」— 一句话价值（如：让 AI 安全读取收益与风控数据、在合规边界内执行投资决策）；列出能力清单与接入步骤。 |

### 3. 交互与交易（真正支持 AI 交易）

| 方向 | 说明 | 可做内容 |
|------|------|----------|
| **只读 vs 可写** | 先只读再可写，降低风险 | 第一阶段：仅查询（池子、APY、LTV、NAV、托管证明）；第二阶段：在合规与 KYC 前提下，支持 Agent 代表已认证用户执行「下单/申购」等，需明确法律责任与确认流程（见下）。 |
| **确认与合规** | AI 执行的是「谁」的操作 | 设计「人类确认」环节：如 Agent 发起订单前需用户一次性确认或授权范围；或仅允许 Agent 在「托管/授权」账户内操作并留痕；避免「纯机器未经授权动用户资金」的合规与信任问题。 |
| **Webhook / 回调** | 让 Agent 被动接收状态变更 | 订阅「池子 APY 变更」「LTV 接近清算线」「订单成交」等事件，推送到 Agent 配置的 URL，便于自动化策略与告警。 |

### 4. 可观测与信任（宣传 + 安全）

| 方向 | 说明 | 可做内容 |
|------|------|----------|
| **用量与监控** | 开发者/机构看得到自己在怎么用 | 控制台：API 调用量、按接口/按 Key 的 QPS、错误率；可选：配额与限速提示、异常登录告警。 |
| **审计与合规** | 证明「谁在何时做了什么」 | 关键操作（下单、改权限、提现）留审计日志；对机构与监管可提供「AI Agent 调用记录」的摘要或导出，便于合规与争议处理。 |
| **沙箱/测试网** | 降低 Agent 试错成本 | 提供 testnet 或模拟环境，Agent 用测试 Key 调用模拟池与模拟订单，不碰真实资金，便于开发与演示。 |

### 5. 对外宣传话术与触点（噱头落地）

| 方向 | 说明 |
|------|------|
| **官网/落地页** | 一句主文案：如「AI-Native RWA Infrastructure」或「Let AI Monitor, Verify & Act on Institutional RWA」；区块：AI Agent Kit、MCP、API 文档入口、用例（如「量化团队用 Agent 自动监控 LTV」）。 |
| **案例与场景** | 抽象 2～3 个场景：例如「AI 监控抵押率并预警」「AI 根据 APY 与风险参数做资产配置建议」「机构用 Agent 统一对接多池数据」— 用于官网、白皮书、演讲。 |
| **合规表述** | 强调「技术能力」：我们提供 API/数据与风控能力，AI 如何用、在何边界内做决策由调用方负责；与页脚免责一致（不直接运营资金、不替用户做投资决策）。 |

## 三、与现有 MRD 的对应

- **F-3.2 AI-Agent Kit & MCP 接入指南**：本文档中的「身份与接入」「协议与文档」「可观测」可直接支撑该需求。
- **F-3.1 智能合约库**：可与「AI 可调用的链上数据」结合宣传（如：Agent 读取合约 NAV/APY）。
- **合规**：AI 相关文案继续避免 Invest/Deposit/Earn，用 Monitor、Verify、Technology、Data、API 等；若涉及「交易」，需明确是「在用户授权与合规前提下的技术能力」。

## 四、小结（供您取舍）

- **必做（您已提）**：API 配置、Key 生成与管理。
- **建议一并考虑**：MCP 文档与示例代码、权限与审计、只读先行再可写、人类确认/授权机制、Webhook、用量与沙箱。
- **宣传**：独立「AI Agent Kit」入口、一句主文案、2～3 个场景化用例，并与合规话术统一。

以上均为思路级记录，具体优先级与实现范围可在设计阶段与老板、技术再对齐。
