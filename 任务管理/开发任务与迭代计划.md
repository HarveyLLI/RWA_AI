# RWA&AI 项目 — 开发任务与迭代计划

> 编制说明：明确从原型到上线的开发顺序；采用「先做通一条主流程，再铺开、再细化」的策略，避免全站半成品。  
> 依据：项目一/项目二信息架构、延伸页面与链接规划、网站开发技术方案与选型。

---

## 一、开发顺序原则

1. **先一条主流程跑通**：在「执行代码」里用选定技术栈实现「从首页到 1～2 个核心页」的完整可访问流程，再扩展其他页面。  
2. **项目一优先**：官网 Web3 Infrastructure 专区为优先交付对象；项目二（基金会产品站）在项目一主流程稳定后再启动。  
3. **原型为蓝本**：开发时以 `原型&UI/官网` 与 `原型&UI/项目二` 的 HTML 为视觉与结构参考，在「执行代码」中独立实现，不直接修改原型 HTML。  
4. **任务可勾选**：下文任务列表可用于日常勾选与进度跟踪（建议复制到本地后自行打勾或更新状态）。

## 十、原型与真实实现的交互差异说明

以下说明**原型 HTML** 与 **执行代码（真实前端应用）** 在「最终可视化界面的交互」上的差异，便于产品与开发对齐预期。

### 10.1 结论概览

| 维度 | 原型 HTML（当前） | 真实代码（执行代码中实现） |
|------|-------------------|----------------------------|
| **视觉与布局** | 可完全一致 | 以原型为蓝本，可做到一致 |
| **连接钱包** | 仅切换 UI（显示/隐藏按钮、假地址如 0x1234…5678），无真实钱包弹窗 | 可接入 MetaMask / WalletConnect 等，真实弹窗、选链、签名，有「绑定钱包」的真实感受 |
| **存入 / 交易** | 点击后弹窗提示「原型演示：实际产品将调用钱包授权并完成存入」 | 真实 approve + 合约 deposit，有 gas、pending、成功/失败状态，用户可见交易确认与链上结果 |
| **持仓 / 数据** | 静态写死（如固定 10,000 wlUSDT-Jun26），不随钱包变化 | 按当前连接地址从链上或后端拉取，换钱包即换持仓 |
| **兑换 $ANTA** | 弹窗「原型演示：兑换流程在实际产品中会调用合约」 | 真实合约调用，含风险确认、交易确认、链上执行 |
| **API Keys** | 可做表单与列表的静态展示 | 可对接后端创建/撤销/鉴权，真实 Key 与权限 |

因此：**原型无法提供「真实绑定钱包」的感受，也无法完成真实交易**；只有在本仓「执行代码」里用选定技术栈实现并接入钱包与合约后，用户才能获得与生产环境一致的交互体验。

### 10.2 对开发与产品协作的约定

1. **原型**：继续作为**视觉与信息结构**的单一来源；对原型的更新（改文案、改布局、改流程示意）不影响执行代码的既有逻辑，但可作为「新一版蓝本」供执行代码对齐。
2. **执行代码**：在实现「连接钱包、存入、赎回、兑换 $ANTA」等能力时，**不依赖原型里的内联 script**（如 `setWalletUI(true)`、`alert(...)`），而是独立实现：钱包 SDK、合约调用、状态与错误处理。
3. **验收**：涉及钱包与交易的页面，验收标准以「在本地/测试网能真实完成一笔操作」为准，而非「与原型点击效果一致」。

### 10.3 项目二相关差异摘要（当前原型已体现）

- **index.html / markets.html**：连接钱包 = 仅前端切换；存入 = 跳转 vault-detail，无链上调用。
- **vault-detail.html**：连接钱包 = `setWalletUI(true)`；存入 = `alert('原型演示：实际产品将调用钱包授权并完成存入')`。
- **投资组合 / 兑换 $ANTA**：持仓为静态数据；兑换 = `alert('原型演示：兑换流程在实际产品中会调用合约')`。

执行代码中实现项目二时，上述能力需全部替换为真实钱包与合约逻辑，界面与文案仍可对照原型保持一致。
